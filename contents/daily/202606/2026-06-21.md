HTTP是Request-Response
Client 發送 Request，Server 回傳 Response 後連線結束。
Server 不會主動推送資料給 Client，如果想要取得最新資料只能一值發Reqeust(Polling)
底層協議: HTTP/TCP/IP

WebSocket 是 Full-Duplex（雙向通訊）。
連線建立後，Client 和 Server 都可以主動傳送資料。
```js
const ws = new WebSocket("ws://localhost:8000")
```
建立成功後：雙方都能主動傳送資料。
建立時先走HTTP (WebSocket Handshake)，後續不再使用 HTTP Request/Response。
```sh
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
```

WebSocket 缺點？
需要維持長連線、Server 要維護大量 Connection、Load Balancer 比較複雜

WebSocket 和 HTTP 差異？
HTTP 採用 Request-Response 模式，由 Client 主動發送請求，Server 回應後連線通常結束；而 WebSocket 會先透過 HTTP Upgrade 建立連線，之後維持長連線，Client 與 Server 都能主動傳送資料，因此適合聊天室、即時通知、股票行情等即時性需求。缺點是 Server 必須維護大量長連線，在多台 Server 部署時通常需要 Redis Pub/Sub 或 Message Broker 來同步訊息。

Q. Connection 如何維持？
WebSocket 底層其實是 TCP。
WebSocket 會保持：Connect -> Send/Recive -> Send/Revice
直到任一方主動關閉

Heartbeat（心跳）
Client ---- ping ----> Server

WebSocket Server 要怎麼管理所有 Connection？

WebSocket 底層是 TCP 長連線，建立後不會像 HTTP 一樣每次請求後關閉。為了避免 NAT、Firewall 或 Load Balancer 將閒置連線回收，通常會透過 Ping/Pong Heartbeat 定期確認連線存活。如果連續幾次未收到 Pong，就判定對方離線並關閉連線。客戶端通常也會實作自動重連機制，例如 Exponential Backoff。若系統是多台 WebSocket Server，則會搭配 Redis Pub/Sub 或 Message Queue 來同步不同節點上的連線狀態與訊息。

TCP（Transmission Control Protocol）是一種可靠的傳輸層協定，負責在兩台主機之間建立連線並傳送資料。它提供順序保證、遺失重傳、錯誤檢查與流量控制，因此應用程式可以像讀寫資料流一樣使用網路。HTTP、HTTPS、WebSocket、SSH 等常見協議都是建立在 TCP 之上。

REST 是以 Resource 為中心的 API 設計風格，透過 URL 表示資源，HTTP Method 表示操作，例如 GET /users/123 或 DELETE /users/123。RPC 則是以 Function 為中心，透過網路呼叫遠端方法，例如 /getUser 或 /transferMoney。REST 通常較適合對外 API，因為易讀且符合 HTTP 語意；RPC 尤其是 gRPC，則常用於微服務間通訊，因為使用 Protocol Buffers，傳輸效率較高、型別安全且能自動產生多語言 Client SDK。

Protocol Buffers（Protobuf）是一種資料序列化格式（Serialization Format），會把資料轉換成緊湊的二進位格式傳輸。
先定義資料結構，再自動轉換成高效率的二進位格式。

Protocol Buffers 的世界

先定義 Schema：
```json
message User {
    int64 id = 1;
    string name = 2;
}
```

Protocol Buffers（Protobuf）是 Google 開發的資料序列化格式。開發者先透過 .proto 檔案定義資料結構與服務介面，接著由 Protobuf Compiler 自動產生程式碼。傳輸時資料會被序列化成緊湊的二進位格式，而不是像 JSON 一樣傳送文字，因此體積更小、解析速度更快。gRPC 通常使用 Protobuf 作為預設資料格式，所以常被一起提到，但 gRPC 是 RPC Framework，而 Protobuf 是資料序列化格式，兩者並不相同。

KVM 和 QEMU 是什麼？
KVM（Kernel-based Virtual Machine）是 Linux Kernel 提供的虛擬化功能，利用 Intel VT-x 或 AMD-V 等硬體虛擬化技術，讓 Guest OS 能以接近原生效能執行。QEMU 則負責模擬虛擬機所需的硬體設備，例如 CPU、磁碟、網卡與記憶體。實務上通常將 KVM 與 QEMU 搭配使用，QEMU 提供虛擬硬體，KVM 負責硬體加速，因此能同時兼顧功能完整性與執行效能。

VM 和 Docker 有什麼差異？
VM 透過 Hypervisor 執行完整 Guest OS，隔離性較高；Docker 則共用 Host Kernel，因此更輕量且啟動更快，但隔離程度相對較低。

Snapshot是什麼 ? 
核心概念：Copy-on-Write（CoW）

Snapshot 是虛擬機在某個時間點的狀態快照，可以快速回復到當時的狀態。它通常不是完整複製 VM，而是利用 Copy-on-Write 技術，只記錄 Snapshot 之後發生變更的資料，因此建立速度很快。

Snapshot 如何實作？
常見做法是 Copy-on-Write。建立 Snapshot 時不複製整顆虛擬磁碟，而是保留原始磁碟作為 Base Image。後續資料修改時，新的 Block 會寫入 Snapshot 檔案中，讀取時優先讀取 Snapshot，再回退到 Base Image。因此建立與 Rollback 都很快，但 Snapshot 過多會增加讀取成本與儲存空間消耗。

Live Migration（熱遷移）是什麼
指的是：在 VM 持續運行的情況下，把 VM 從一台實體主機搬到另一台實體主機。

最常見做法: Pre-Copy Migration
STEP1: VM 繼續執行,同時開始複製 RAM,使用者持續操作RAM被改變(Dirty Pages)
STEP2: 同步Dirty Pages
STEP3: 停機，搬最後資料CPU Registers,MEMORY，切換Server，恢復VM
